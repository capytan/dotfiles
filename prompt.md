# dotfiles の Claude Code 設定 全面レビュー依頼

このリポジトリ（`/Users/capytan/dotfiles`）で管理している **Claude Code 全般の設定**を全面レビューしてください。目的は、設定の正確性・整合性・保守性を高め、あわせて日常的に使うモデル（Claude Opus 4.8 / Claude Sonnet 5）に対して最適化されているかを検証することです。

まず要求を確定させてから動いてください。曖昧さが残る点は、根拠のない仮定で進めず、レビュー計画を立てる前に短く確認してください。逆に、コードやファイルから判断できることは自分で確かめて進めてください。

## 前提（常用モデル）

- この設定は **Opus 4.8 と Sonnet 5 を日常のドライバーとして** Claude Code を動かす前提で最適化したい。Fable 5 は時々使う程度。
- したがってレビューは **Opus 4.8 / Sonnet 5 を主対象**に評価してください。Fable 5 固有の挙動（`reasoning_extraction` refusal、adaptive thinking、effort 依存の冗長化など）は「Fable 5 を使うセッションに限って影響する副次論点」として扱い、主軸には置かないでください。
- モデルは shell の alias で切り替えて併用しています（3モデルを行き来する）。`configs/claude/settings.json` の設定は**モデルを跨いだ単一のグローバル値**として全モデルに一律で効く、という点を評価の前提にしてください。

## レビュー対象（スコープ）

主対象は Claude Code 設定一式です。

- `configs/claude/CLAUDE.md` — グローバルルール（`~/.claude/CLAUDE.md` にシンボリックリンク）
- `configs/claude/settings.json` / `configs/claude/mcp.json` — 設定・MCP
- `configs/claude/statusline.sh` / `configs/claude/setup-claude.sh` — スクリプト
- `configs/claude/agents/**` — カスタム subagent 定義（14個）
- `configs/claude/hooks/**` — hook スクリプト群（PreToolUse バリデータ、tmux 状態表示など）
- `configs/claude/skills/**` — カスタム skill 群（レビュー系・日本語ライティング系・意思決定支援など）
- `.claude/rules/**` — パススコープされたリポジトリ内ルール
- リポジトリ直下の `CLAUDE.md` — このリポジトリ用のプロジェクトルール（`~/.claude/CLAUDE.md` とはスコープが別）

参考として、設定がどう配置・symlink されるかは `install.sh` と `configs/claude/setup-claude.sh` を確認してください。`shell/`・`platform/` など Claude Code 以外の dotfiles 本体はスコープ外です（ただし Claude 設定と相互参照している箇所があれば指摘してよい）。

## レビュー観点

以下を軸に、重要なものから見てください。網羅チェックリストの消化ではなく、実際に効いてくる問題の発見を優先してください。

1. **正確性・整合性**
   - ファイル・パス・シンボル・コマンドの参照が実在するか（CLAUDE.md や rules が言及するファイル/関数/フラグが現存するか）
   - `settings.json` / `mcp.json` の構文と스키마妥当性、hook のコマンドパスが実体と一致するか
   - ドキュメントの記述と実装の乖離（説明されている挙動と実際のスクリプトの挙動のズレ）
   - agents/skills の frontmatter（name, description, tools, model 等）の妥当性とトリガー条件の質

2. **現行モデル（Opus 4.8 / Sonnet 5）への適合性**（このレビューの主眼のひとつ）
   - 旧モデル向けに過度に規範的・冗長な skill / prompt がないか。現行モデルは指示追従が強く、細かい列挙より簡潔な指示のほうが機能する。弱いプランニングを補うために書かれた過剰な手順指示は、むしろ default の品質を落としうる。削除・簡素化できるものを洗い出すこと
   - 能力向上に伴い**もはや不要になったガードレール・ツール制約・防御的指示**。旧モデル前提の「安全策」が残って、今はノイズになっていないか
   - **ユーザーの方針との矛盾**。グローバル `CLAUDE.md` は「冗長な前置きは不要、直接本題へ」と簡潔さを求めている。これと逆方向に働く設定がないか。具体例として `settings.json` の `outputStyle`（例: `Explanatory`）は、モデルを跨ぐ**単一のグローバル値**として system prompt を書き換え、説明を付加して応答を長くする。Opus 4.8 / Sonnet 5 では refusal リスクはなく解説の実用価値もあるが、グローバル既定にすると常時冗長化がかかり、上記の簡潔化方針と綱引きになる。この設定を「グローバル既定に置くべきか / セッション単位に留めるべきか」を評価すること。加えて、この機能自体が本家で plugin（SessionStart hook）へ移行し deprecated 扱いのシグナルがある点も、committed dotfiles の脆さとして考慮に入れてよい
   - （副次・Fable 5 を使うときだけ関係）内部推論を応答本文へ再現・転記・説明させる指示は、Fable 5 で `reasoning_extraction` refusal を誘発しうる。常用が Opus 4.8 / Sonnet 5 なら実害は限定的だが、skill / agent / CLAUDE.md に該当する記述があれば軽く指摘しておくこと

3. **冗長・重複・矛盾**
   - グローバル CLAUDE.md／プロジェクト CLAUDE.md／rules／skills 間の重複や矛盾した指示
   - 同じことを複数箇所で規定していて、片方だけ更新されると壊れる箇所
   - 既に多数のレビュー系 skill（`claude-config-reviewer`, `auditing-claude-config`, `agent-team-review`, `codebase-review`, `prompt-review`）が存在する。役割の重複・棲み分けの明確さも評価対象

4. **hooks の堅牢性・安全性**
   - `pretooluse-validate-command.sh` の危険パターン検知（force-push, reset --hard, rm -rf, 単独 sed/awk 等）に抜け穴や誤検知がないか
   - tmux 状態表示 hook 群の失敗時挙動（hook の失敗が本来の作業をブロックしないか）、シェルの堅牢性（クォート、`set -e`、未定義変数）
   - hook が前提とする環境変数・キャッシュファイル・パスの前提が壊れやすくないか

5. **保守性・一貫性**
   - 命名・構成規約の一貫性（このリポジトリ既存のスタイルに沿っているか）
   - `configs/` / `shell/` / `platform/` の三層分離、ローカルファイル（`settings.local.json` 等をコミットしない）の原則が守られているか

## 進め方

- **subagent を積極活用**してください。観点や対象ディレクトリごとに独立レビューを並列で走らせ、非同期に進めて構いません。独立したサブタスクは委譲し、脱線や文脈不足があれば介入してください。
- リポジトリ内の既存レビュー系 skill / agent（上記）を、使えるなら道具として活用してよいです。同時に、それら自体もレビュー対象です。
- **進捗や結論は、必ずこのセッションのツール結果を根拠に報告してください。** 未検証のことは「未検証」と明示し、確認できた指摘だけを断定してください。テストや検証をしていない箇所を「問題なし」と書かないこと。
- 指摘には必ず `file_path:line` を添え、なぜ問題か・どう直すかを簡潔に書いてください。
- タスクの範囲を超えた整形・リファクタ・抽象化の提案は不要です。今回は**発見の報告が成果物**であり、勝手にファイルを修正しないでください。修正は、レポート提示後にこちらが依頼してから着手してください。

## 成果物

日本語で、次の構成のレビューレポートを提示してください。

1. **要約（TLDR）** — 最初の数文で「全体として何が問題か / 何が良いか」。読み手が最初に知りたい結論を先に。
2. **重大度別の指摘一覧** — Critical / Warning / Nits に分け、各項目に `file_path:line`・問題・推奨対応。重大なものから。
3. **現行モデル適合性の所見** — 観点2に対する総括（削除・簡素化できる指示、Opus 4.8 / Sonnet 5 で不要になったガードレール、ユーザーの簡潔化方針と矛盾する設定、outputStyle の是非）。Fable 5 固有の論点はここで副次的に触れる程度に。
4. **重複・矛盾マップ** — CLAUDE.md / rules / skills 間で整理すべき箇所。
5. **推奨アクション** — 優先順位付き。すぐ直すべきもの / 検討の余地があるもの を分けて。
