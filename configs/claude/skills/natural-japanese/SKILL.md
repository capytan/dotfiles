# Natural Japanese — AI的日本語を自然な日本語に変換する

## このスキルの目的

生成AIが出力する日本語には、人間が書かない特有のパターンがある。
これらは文法的に正しくても「読むと違和感がある」「AI臭い」と感じさせる。
このスキルは、そうしたパターンを体系的に検出・変換し、
読み手にとって自然で、書き手の意図が伝わる日本語に仕上げる。

## 変換の原則

1. **削れるものは削る** — 冗長な表現は短くする。意味が変わらないなら短い方が良い
2. **具体的に言い換える** — 抽象語を具体的な動詞・名詞に置き換える
3. **リズムを作る** — 語尾や文の長さに変化をつけて、読みやすくする
4. **書き手の声を残す** — 過度に書き換えない。元の意図・トーンを尊重する
5. **消した後に声を入れる** — パターン除去だけで終わらない。体験・意見・体温を足す

## 検出カテゴリ A〜I

詳細な検出パターン表・変換例・クイック変換表は [references/patterns.md](references/patterns.md) を参照する。
スキャン時は以下の順で当該リファレンスのセクションを開く。

- **A. 語彙レベル** — AI頻出ワード（「これにより」「〜することができます」「活用する」「〜において」「〜する必要があります」「大幅に」等）を文脈に合った具体語/和語へ置換、または削除。抽象動詞・抽象形容詞・意義の過剰強調・定型評価語（「浮き彫り」「今後の展開」「注目に値する」等）も対象
- **B. 構造レベル** — 単調な語尾、接続詞過多（「まず／次に／最後に」「さらに／また／加えて」「一方で／しかしながら」の連続）、全角ダッシュ（——）、三点セット強制、否定並列構文、主語の過剰明示、300字以下のセクションへの小見出し
- **C. 意味レベル** — 安全クッション表現（「一般的に」「多くの場合」）、受動態の婉曲化、データ後の付け足し構文（「〜を示しており」「〜を物語っています」）、曖昧な出典
- **D. AI的な装飾** — 比喩（羅針盤/エンジン/両輪/架け橋/DNA/レシピ等）の濫用。1文章で1-2個まで
- **E. 翻訳調** — カタカナ語の3語以上の連続、和語で言えるものを英語直輸入で書く（ソリューション/インサイト/イニシアチブ/ジャーニー/ベストプラクティス等）
- **F. テンプレ感** — 「本記事では〜解説します」「いかがでしたか？」「STEP 1:」装飾、追従的トーン（「素晴らしいご質問ですね！」）、説教前置き（「ここで重要なのは〜」）
- **G. Markdown構造のAI臭さ** — リスト項目全てが太字+コロン（`**項目**: 説明`）、絵文字プレフィックス（✅❌💡🚀等）、述語+コロン+ブロック要素、見出し内の太字（textlint-rule-preset-ai-writing の `no-ai-list-formatting` / `no-ai-emphasis-patterns` / `no-ai-colon-continuation` 準拠）
- **H. 文体の統一** — 「です・ます」と「だ・である」の混在、カジュアル度の混在、敬語レベル不統一、一人称混在（「私」「弊社」「我々」「当社」）
- **I. 声を入れる** — パターン除去後の空っぽさを補う。体験を1つ入れる/意見を1つ入れる/リズム崩し/不完全さを認める/体温のある結論

文体の判断基準（H）:
- **フォーマルビジネス**: 「です・ます」+ 敬語。「〜いたします」「〜ございます」
- **一般的なブログ**: 「です・ます」（ライト）。「〜です」「〜ますね」「〜でしょう」
- **テックブログ**: 「だ・である」調が主流。「〜してみた」「〜だと思う」
- **SNS/カジュアル**: 「〜だよ」「〜じゃん」「〜かも」

ユーザー指定がない場合は、入力テキストの多数派の文体に合わせる。

## 実行手順

1. **入力テキストを通読する** — 全体のトーン・目的・読者を把握する
2. **文体を判定する (H)** — 「です・ます」か「だ・である」か、フォーマル度を確認。ユーザー指定があればそれに従う
3. **パターンA〜Gを順にスキャンする** — `references/patterns.md` の対応セクションを参照し、該当箇所をマークする
4. **変換を適用する** — 元の意図を壊さないよう注意しながら書き換える
5. **声を入れる (I)** — パターンを消した箇所に、体験・意見・リズムの変化を補う
6. **文体の統一を確認する (H)** — 変換後に文体の混在が生じていないか最終チェック
7. **二重チェック** — 書き換え後の文章を読み返し「AIが書いたと感じないか？」を自問する。感じたら該当箇所を再度書き換える
8. **変更箇所を提示する** — 元の表現と変換後の表現を対比して見せる

**エッジケース:**
- AI的パターンが見つからない場合 → 変更不要と伝え、理由を添える
- 1〜2文の短い入力 → 変更点の表のみ提示。全文掲載は不要
- 法務文書・規約など意図的にフォーマルな文体 → このスキルの対象外。ユーザーに確認する

## 出力フォーマット

変換結果は以下の形式で提示する:

```
## リライト結果

（変換後の全文）

## 主な変更点

| 変更前 | 変更後 | 理由 |
|---|---|---|
| これにより、生産性が向上します | そのため、作業時間が半分になります | 抽象→具体、AI頻出語の置換 |
```

変更が少ない場合は変更点の表だけでもよい。変更が多い場合はカテゴリ別にまとめてもよい。
