# Natural Japanese — AI的日本語を自然な日本語に変換する

## このスキルの目的

生成AIが出力する日本語には、人間が書かない特有のパターンがある。
これらは文法的に正しくても「読むと違和感がある」「AI臭い」と感じさせる。
このスキルは、そうしたパターンを体系的に検出・変換し、
読み手にとって自然で、書き手の意図が伝わる日本語に仕上げる。

## 変換の原則

1. **削れるものは削る** — 冗長な表現は短くする。意味が変わらないなら短い方が良い
2. **具体的に言い換える** — 抽象語を具体的な動詞・名詞に置き換える
3. **リズムを作る** — 語尾や文の長さに変化をつけて、読みやすくする
4. **書き手の声を残す** — 過度に書き換えない。元の意図・トーンを尊重する

## AI的表現の検出と変換パターン

以下のカテゴリ順にチェックする。詳細な変換例は `references/patterns.md` を参照。

### A. 語彙レベル — AI頻出ワードの置き換え

| 検出パターン                 | 変換方針                                                             |
| ---------------------------- | -------------------------------------------------------------------- |
| 「これにより」               | 「そのため」「その結果」「おかげで」など文脈に合う接続に。または削除 |
| 「〜することができます」     | 「〜できます」                                                       |
| 「〜において」               | 「〜で」「〜では」                                                   |
| 「〜を実現します」           | 具体的に何をするか書く                                               |
| 「〜の最適化」「最適な」     | 具体的に何が良くなるか書く                                           |
| 「本質的な」「革新的な」     | 削除するか、具体的な説明に置き換え                                   |
| 「効果的な」「重要な」       | 何がどう効くのか具体的に                                             |
| 「活用する」「leverageする」 | 「使う」「取り入れる」                                               |
| 「提供する」「サポートする」 | 具体的な動作に（「教える」「手伝う」「届ける」等）                   |
| 「確保する」                 | 「取る」「用意する」「押さえる」                                     |
| 「促進する」                 | 「進める」「後押しする」                                             |
| 「〜する必要があります」     | 「〜しなければならない」「〜すべきだ」（または削除して直接書く）      |
| 「まず最初に」               | 「まず」（重複削除）                                                 |
| 「大幅に」「完全に」「非常に」 | 定量的な表現に（「30%減った」「全件対応した」等）。無理なら削除     |
| 「必要に応じて」             | どんな場合に必要か具体的に書く。書けないなら削除                     |

### B. 構造レベル — 文のリズムと接続

| 検出パターン                       | 変換方針                                                   |
| ---------------------------------- | ---------------------------------------------------------- |
| 「〜です。〜です。〜です。」連続   | 体言止め、「〜だろう」「〜かもしれない」「〜だ」等を混ぜる |
| 「さらに」「また」「加えて」の連続 | 接続詞を削って文の流れで繋げる。全体の半分以下に減らす     |
| 「まず」「次に」「最後に」の列挙   | 必要な場合のみ残す。読み手が自然に順序を追える場合は削除   |
| 「一方で」「しかしながら」         | 「ただ」「でも」「けれど」など文体に合うものに             |
| 段落冒頭が毎回接続詞で始まる       | 接続詞なしで始まる段落を増やす                             |
| 全ての文がほぼ同じ長さ             | 短い文（10字以下）と長い文を意図的に混ぜる                 |

### C. 意味レベル — 抽象度と具体性

| 検出パターン                   | 変換方針                                   |
| ------------------------------ | ------------------------------------------ |
| 「一般的に〜」「多くの場合〜」 | 削除するか、具体的な条件に置き換え         |
| 「〜が期待されています」       | 誰が期待しているのか明示するか、断言に変更 |
| 「状況によって異なります」     | どんな状況でどう異なるのか具体的に         |
| 「本記事では〜を解説します」   | 削除（読めばわかる）                       |
| 「結論から言うと」             | 削除して結論をそのまま書く                 |
| 「〜と言えるでしょう」         | 断言できるなら断言する                     |

### D. AI的な装飾 — 比喩・修辞の問題

| 検出パターン                     | 変換方針                                             |
| -------------------------------- | ---------------------------------------------------- |
| 「羅針盤」「コンパス」「道標」   | 具体的に何を指すのか直接書く                         |
| 「エンジン」「原動力」「推進力」 | 同上                                                 |
| 「両輪」「土台」「柱」「基盤」   | 同上                                                 |
| 「架け橋」「潤滑油」「スパイス」 | 同上                                                 |
| 「DNA」「レシピ」                | 同上                                                 |
| 同じ比喩の使い回し               | 1つの文章で比喩は1-2個まで。それ以上は直接的な表現に |

### E. 翻訳調・カタカナ語の不自然さ

| 検出パターン                     | 変換方針                                     |
| -------------------------------- | -------------------------------------------- |
| 「フリーランサー」               | 「フリーランス」（日本語圏の慣用に合わせる） |
| 「デーティングアプリ」           | 「マッチングアプリ」                         |
| 不要なカタカナ語                 | 日本語で自然に言える場合はそちらを使う       |
| コロン後の半角スペース（英語式） | 日本語の慣習に合わせる                       |
| Markdown記法の残存（**太字**等） | プレーンテキストの場合は削除                 |

### F. テンプレ感・説明書感

| 検出パターン                       | 変換方針                                     |
| ---------------------------------- | -------------------------------------------- |
| 「以下で詳しく解説します」         | 削除（前置きせず書き始める）                 |
| 「ステップ1:」「STEP 2:」          | 内容に応じて、番号付きリストか見出しか段落に |
| 「いかがでしたか？」（まとめ冒頭） | 削除                                         |
| 「〜を理解することが重要です」     | 理解すべき内容を直接書く                     |
| 冒頭と末尾で同じ内容を繰り返す     | 片方を削除                                   |

### G. Markdown構造のAI臭さ（textlint-rule-preset-ai-writing 準拠）

Markdown文書の場合に追加でチェックする。プレーンテキストでは不要。

| 検出パターン                                  | 変換方針                                           |
| --------------------------------------------- | -------------------------------------------------- |
| リスト項目が全て太字+コロン（`**項目**: 説明`）| 太字を外して普通のリスト項目にする                  |
| リスト項目が全て絵文字プレフィックス（✅❌💡🚀等）| 絵文字を外す                                  |
| 述語の後にコロン+ブロック要素（`実行します:`）  | コロンを句点に変えるか、文を名詞で終える           |
| 見出し内の太字（`## **重要な**設定`）           | 見出し自体が強調なので太字は不要                   |
| 絵文字+太字の組み合わせ強調                     | どちらか一方に。基本は両方外す                     |

### H. 文体の統一

文体（「です・ます」調 / 「だ・である」調）の混在はAI感を強く与える。
ユーザーが指定したトーンに合わせて、全体の文体を統一する。

| 検出パターン | 変換方針 |
|---|---|
| 「です・ます」調と「だ・である」調の混在 | ユーザー指定のトーンに統一。指定がなければ元テキストの多数派に合わせる |
| カジュアル表現（「〜してない？」「〜じゃん」）と丁寧語の混在 | カジュアルに寄せるか丁寧に寄せるか統一する |
| 敬語レベルの不統一（「です」と「ございます」の混在） | 読者との距離感に合わせて統一 |
| 一人称の混在（「私」「弊社」「我々」「当社」） | 1つに統一 |

文体の判断基準:
- **フォーマルビジネス**: 「です・ます」+ 敬語。「〜いたします」「〜ございます」
- **社内ブログ/カジュアル**: 「だ・である」調、または「です・ます」のライトな使い方
- **テックブログ**: 「だ・である」調が主流。「〜してみた」「〜だと思う」

## 実行手順

1. **入力テキストを通読する** — 全体のトーン・目的・読者を把握する
2. **文体を判定する (H)** — 「です・ます」か「だ・である」か、フォーマル度を確認。ユーザー指定があればそれに従う
3. **パターンA〜Gを順にスキャンする** — 該当箇所をマークする
4. **変換を適用する** — 元の意図を壊さないよう注意しながら書き換える
5. **文体の統一を確認する (H)** — 変換後に文体の混在が生じていないか最終チェック
6. **通読して確認する** — 変換後の文章が自然に読めるか、リズムに変化があるかチェック
7. **変更箇所を提示する** — 元の表現と変換後の表現を対比して見せる

## 出力フォーマット

変換結果は以下の形式で提示する:

```
## リライト結果

（変換後の全文）

## 主な変更点

| 変更前 | 変更後 | 理由 |
|---|---|---|
| これにより、生産性が向上します | そのため、作業時間が半分になります | 抽象→具体、AI頻出語の置換 |
```

変更が少ない場合は変更点の表だけでもよい。
変更が多い場合は、カテゴリ別にまとめてもよい。
